# 要件定義書 — Serverless Image Resize API

## 1. プロジェクト概要

画像のアップロード・自動リサイズ・CDN配信を行うサーバーレス API。
個人ポートフォリオとして、AWS サーバーレスアーキテクチャの設計・構築・運用スキルを実証することを目的とする。

## 2. 機能要件

### FR-1: 画像アップロード

- REST API 経由で画像ファイル（JPEG, PNG, WebP）をアップロードできる
- アップロード上限: 10MB/ファイル
- アップロード完了時に一意の画像 ID を返却する

### FR-2: 自動リサイズ

- アップロードされた画像を以下の3サイズに自動リサイズする
  - Thumbnail: 150x150px（アスペクト比維持、クロップ）
  - Medium: 800px幅（アスペクト比維持）
  - Large: 1920px幅（アスペクト比維持）
- 元画像より大きいサイズへのリサイズは行わない（拡大防止）

### FR-3: 画像取得

- 画像 ID を指定して、メタデータと配信 URL を取得できる
- 配信 URL は CDN 経由の署名付き URL とする

### FR-4: 画像削除

- 画像 ID を指定して、元画像・リサイズ済み画像・メタデータを一括削除できる

## 3. 非機能要件

### NFR-1: 可用性

| 項目 | 目標値 | 理由 |
|------|--------|------|
| API 稼働率 | 99.9% | サーバーレスのマネージド SLA に準拠 |
| データ耐久性 | 99.999999999% (11 nines) | S3 Standard の標準耐久性 |

**Why**: 個人ポートフォリオとして、本番同等の可用性設計思考を示すため。実運用では SLA を下げることでコスト削減も可能。

### NFR-2: パフォーマンス

| 項目 | 目標値 | 理由 |
|------|--------|------|
| API レスポンス（アップロード） | < 3秒 | 10MBファイルの転送時間を考慮 |
| API レスポンス（取得） | < 200ms | DynamoDB の低レイテンシを活用 |
| リサイズ処理完了 | < 10秒 | Lambda のメモリ512MB想定 |
| CDN 配信レイテンシ | < 50ms | CloudFront エッジキャッシュ |

### NFR-3: セキュリティ

| 項目 | 実装方針 |
|------|----------|
| 認証 | API Key（API Gateway Usage Plan） |
| 通信暗号化 | HTTPS のみ（TLS 1.2+） |
| 保存データ暗号化 | S3 デフォルト暗号化（SSE-S3） |
| IAM | 最小権限の原則（Lambda ごとに個別ロール） |
| S3 公開設定 | パブリックアクセス全ブロック |

**Why**: セキュリティ設計は面接で必ず聞かれる。「なぜその設定か」を説明できるようにするため、設計段階で明文化する。

### NFR-4: コスト

| 項目 | 目標値 | 理由 |
|------|--------|------|
| 月額運用費 | < $3 | 個人利用（月1,000リクエスト想定） |
| 固定費 | $0 | NAT Gateway 等の固定費サービス不使用 |

**Why**: コスト意識は実務で最重要スキルの一つ。無料枠を理解し活用できることを示す。

### NFR-5: 運用性

| 項目 | 実装方針 |
|------|----------|
| IaC | Terraform でインフラ全体を管理 |
| CI/CD | GitHub Actions（PR で plan、merge で apply） |
| ログ | CloudWatch Logs（Lambda 自動出力） |
| モニタリング | CloudWatch Metrics + Alarms |
| コスト監視 | AWS Budgets アラート（$5/月） |
| リソースタグ | 全リソースに Project, Environment, ManagedBy タグ |

## 4. 制約事項

- AWS 無料枠の範囲内で構築することを優先する
- VPC は使用しない（Lambda は VPC 外で実行し、NAT Gateway コストを回避）
- マルチリージョン構成は対象外（CloudFront でグローバル配信は実現）
- ユーザー管理機能は MVP スコープ外（API Key による簡易認証のみ）
